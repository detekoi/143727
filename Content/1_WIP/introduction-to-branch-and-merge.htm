Introduction to branching and merging

The following was moved from then compare-merge-branches page - it needs to be reviewed and perhaps rewritten as a standalone topic.

provides branches as a way to manage multiple versions of a baseline that correspond to multiple versions of an application. You use the features on this page to merge a source branch into a target branch.

Typically, you will use baseline branches when you have multiple versions of an application that have a common set of tests. For example you may have Release and Beta versions of your application, or you may develop each feature in a separate branch and then merge then into a main branch when they are ready. Regression tests that test existing features are common to all versions.

branches allows you to:

  • Create a new baseline branch, based on an existing parent branch, so that regressions tests can be executed on existing features that are not expected to change. From day 1 when you start to work on a new feature, will compare the current visual behavior of the system to the expected behavior expressed by the baseline, detect and report differences, and allow you to decide if the difference is because of an unexpected impact on existing behavior, or because of new behavior that requires a change in the testing.
  • Update the new branch with new tests and checkpoints, and change or delete existing tests and checkpoints to take into account the new behavior introduced by a new feature.
  • When development of the new feature is complete, including visual UI testing, merge the branch baselines back into the parent branch so that the parent branch will have an up-to-date baseline that defines what the expected result should be after the software is merged.
  • Since the new version may clash in some way with the parent version, especially if in the mean time other changes have been introduced in to the parent branch, provides conflict detection and both automatic and manual resolution.

In summary, not only verifies the correctness of each version of your software, it also verifies that what is common in different versions is tested against the same baseline, detects visual clashes between versions you may not have not foreseen, and validates that the merged version of the software meets the expectations expressed by the baselines of both the source and target branches.

The compare and merge branches page of the is the tool to use for this final step where you want to merge the baseline information of your working branch of the new feature (the source branch) into the main development branch (the target branch).

Workflow

The baseline for a test is defined by the application and test names and by the execution environment. A variant of this baseline that is specific for a given version of the application software is defined by defined the branch name in your test code using the  setBranch() method. will automatically find and use the baseline variant for that branch, creating if it does not yet exist. Working with under a given branch proceeds as normal, You run tests, detects differences and you decide if these differences are due to regressions in the application or tests, or bugs in new features or tests. You then update the baseline - these update are saved on the same branch that the test ran on.
