Reviewing test results and updating the baseline

After a test run completes, a team member will typically want to review the results to see which checkpoints matched baseline images and which did not, and to take any necessary corrective action. The Eyes Test Manager provides tools to do the following tasks:

  • View the match results for every step in both detail and summary views.
  • Decide if checks have failed or passed.
  • If a mismatch was caused by an out-of-date baseline image, update the baseline image with the screenshot from the test.
  • Remove baseline images that no longer correspond to an existing checkpoint and add baseline images for new checkpoints.
  • Report and assign bugs to team members.
  • Deal with special cases such as dynamic text such as time of day, which is different on every run, and can cause mismatch detection.

Accessing the Eyes Test Manager

To see the results, log in to the Applitools cloud by entering the following link into your browser:

https://applitools.com/users/login

You will be prompted for your username and password. After you enter these, the browser will switch to the Eyes Test Manager.

Test manager screen overview

The Eyes Test Manager screen is divided into two main areas. In the left panel, Test Manager displays information about the batches that were run by your team and allows you to select the “current” batch to review. On the right, Test Manager displays the tests and steps of the batch you selected. More specifically:

  • The batch list panel on the left displays the most recently run batches. Clicking on any entry in the list will make that batch the "current batch".
  • The batch management panel above the batch list provides tools to update, filter and delete entries in the batch list.
  • The main panel displays the tests in the current batch, and the steps in each test.
  • In the toolbar above the main panel, Test Manager provides buttons that allow you to select from a number of different views of this information. Each view displays a different subset of the available information this allows you to focus on the information you need for your current task. Other tools are provided to delete tests, assign tests to team members, filter tests and steps, group tests by differences and more.
  • Above the main panel, Test Manager provides a summary of the results of the current batch and the progress of your review.

Reviewing and resolving a test run

The Test Manager provides multiple ways of viewing the test and step results; each such view focuses on different aspects of the results. See  Test Manager Views. In the description that follows, we will use the “Test details” view. You choose this view by pressing the button . Initially, you will see a screen similar to the one below with a list of the tests in the batch.

If you click on any of the tests, then it will expand, and you will see a series of thumbnail images that represent the “steps” of the test.

What is a “step"?

A step corresponds to a checkpoint in the test code. Eyes creates a step in the results for every new checkpoint that it discovers. If you remove a checkpoint in the code, the baseline image that corresponded to the checkpoint still exists in the baseline checkpoint. So, when you view the results you will see steps that correspond to a baseline image and either match or don’t match, and you will also see steps that represent new checkpoints without baseline images or baseline images of deleted checkpoints. The small icons at the top left of each step indicate if there was a mismatch and if there was – what type (from left to right above: match, missing checkpoint, mismatch, new checkpoint).

When a Tester reviews the results, he must resolve two types of step mismatches:

  • Steps whose baseline image were paired with a checkpoint but did not match visually, i.e., there is a visual mismatch.
  • Steps for checkpoints without a corresponding baseline image or baseline images without a corresponding checkpoint image.

Accepting and rejecting steps.

Eyes provides tools to either “Accept” or “Reject” a step. These icons appear when you put the cursor over a step thumbnail. We will explain below what exactly accept and reject mean in the context of a particular step result, but in general, they impact two things, the first is if the step is set to "passed" or "failed" and the second is if the baseline image should be replaced by the new checkpoint image or not.

Note that in addition to the accept and reject commands, some other operations are also available. For more information see Using the Step Thumbnail Commands.

Before going into more details on what accept and reject do, lets first understand the difference in workflow between the very first run of a test and subsequent runs of that test.

Example: reviewing a test run

The workflow for handling test results the first time you run a test is different from the workflow on subsequent runs. The first time a test is run, there are no baseline images, all the checkpoints are new, and they are automatically set to be the sequence of baseline image. Apart from reviewing the checkpoint images, the Tester is not required to resolve anything. In subsequent runs, the tester has to analyze and resolve all the mismatches.

Reviewing the first run

After the test is run for the first time, it’s status is "new". The screenshot below shows what this looks like in the Eyes Test Manager. The label to the left of the test name indicates the test is new.

The icon in the top left corner of the step thumbnail indicates that the step is a new checkpoint. In a new test, all the checkpoints are assumed to be correct, and the steps have a passed status as indicated by the green bar.

The icon underneath the indicates that the checkpoint image captured during the test was accepted as a new baseline image.

On the first run of a test, when there is no existing baseline, by default Eyes automatically adds the checkpoint image to the baseline. This means that in this case, the Tester does not need to take any further action. However, it is good practice for a Tester to review the step images and ensure they look as expected. If required by your workflow, you can change the default via the SDK save new tests method. Reviewing subsequent runs

After the first test run, Eyes will have automatically created a baseline with all the checkpoint images captured during that first run. In subsequent test runs, Eyes compares the sequence of captured images to the sequence of images in the baseline, and there are four possible outcomes, as you can see in the following screenshot of a test result:

Note the four different symbols at the top left of each step thumbnail, and the green or orange bars to the left of the thumbnail. The green bar indicates that the test has passed, the orange bar indicates that the Tester is required to resolve the differences, i.e., to investigate the cause of the mismatch and what to do about it.

The four possible match results when there is an existing baseline are as follows:

The checkpoint image matches the baseline image. In most cases this is the expected result, hence the default green “pass” status. However, in some rare cases, a match can indicate a bug. An example of this is if a new feature was supposed to be displayed but is missing in the UI. In this case, the Tester should reject the step and report the issue. If he does so, then the step will be marked as failed (a red bar) and the baseline image will not be changed. On the next run, after the UI is fixed, the difference will be found, and the Tester can accept the image so that it is updated in the baseline.

Eyes did not find a checkpoint image that corresponds to an existing baseline image. If the checkpoint was intentionally removed from the test, then the Tester should accept the step and Test Manager will remove the image from the baseline and pass the step. If this is not intentional, then the Tester can report a bug (in this case, presumably the bug is in the test and not the application).

The checkpoint image does not match the corresponding baseline image. The pink areas in the thumbnail indicate regions where differences were detected. Differences may arise for different reasons, and it is up to the Tester to analyze the differences using the tools that Test Manager provides and to take appropriate actions. Typical scenarios are:

  1. The difference is due to an expected change in the application (e.g., a new feature) and the tester accepts it so that the checkpoint image will replace the obsolete baseline image.
  2. The difference is due to a bug in the application, or perhaps the test, in which case the tester can reject it and reports an issue.
  3. The mismatches are artifacts of the test implementation (for example time of day display that is dynamic and changes on every run), in which case the Tester can choose to accept the step, so that is not marked as a failure and take steps to eliminate or ignore the artifacts. See TBDREF for a discussion of the possible steps he can take.

There is a new checkpoint image with no corresponding baseline image. Typically, this is because of a new checkpoint in which case the Tester will accept the Step. Otherwise, he can reject it and report a bug (probably in the test code and not in the application). Note that we previously saw this icon in the case of a new test with a green bar. In the case of the first run of a test, all the steps are automatically accepted. In the case of a test with an existing baseline, new steps need to be resolved.

Accept and reject status

After the user presses the accept or reject buttons, icons appear on the step thumbnail that indicates if accept or reject were pressed. The status of the test is indicated by the color of the bar on the left – a green bar means that the step passed and a red bar means that the test failed. Eyes keeps account which steps have been reviewed (accepted or rejected) and which haven't, and this determines if the test is considered resolved and fully reviewed or not.

Remember that only in the case of a new test, Eyes displays the accept icon automatically, to indicate that Eyes has automatically accepted the step.

The test and batch status

In addition to the step status that changes when the accept and reject buttons are pressed, Eyes also displays the status of each test and the overall status of the batch. These are updated automatically to reflect the overall state of the test and the batch.

Test status

In general, the status of the test has one of the values in the table below.

Depending on the circumstances, the color of the icon may change, and there may be additional icons next to it. The possible values are fully described in Test and Batch Status.

The test is running

If the test was aborted before it completed

If any of the test steps is unresolved (i.e., there was a mismatch that was not accepted or rejected)

If there are no unresolved steps in the test and one or more steps failed

If all of the steps in the test passed (either by default or explicitly by the Tester)

Overriding the default test status

You can override the default test status by clicking the status label to open the “Override test status” dialog.

Choose from one of the following three options:

Force the test to be “Passed” irrespective of the pass/fail status of its steps.

Force the test to be “Failed” irrespective of the pass/fail status of its steps.

Test Manager should infer the pass/fail status of the test from the state of its steps.

Saving the baseline

Any changes you make to test steps by accepting and rejecting them or by adding or removing annotations are automatically persisted. However, your changes will only apply to subsequent test runs after you explicitly save them to the baseline using the button.

The Test Manager indicates that there are unsaved changes in several ways:

  1. For the step as an asterisk (*) to the right of the step name
  2. For the test as an asterisk (*) to the right of the test name
  3. For the current Batch - an indication of the number of unsaved steps in all the tests of the batch.
  4. For a batch in the batch list - the "Unsaved" label to the right of the batch status
  5. The icon "pulses".

When you hover the cursor over the save button you are offered the following choices (arrow one below):

  • Save changes to baseline. Choosing this option will save all the unsaved changes in the batch.
  • Save selected tests to baseline. Choosing this option only saves the test(s) whose checkbox has been checked even if they were not changed (arrow two below).
