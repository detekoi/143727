Merging branches

This article provides you with information that may be helpful to you for understanding what options you have when merging two branches in . The description below first defines the basic rules that define Eyes behavior, and then summarizes the possible actions you can take in various circumstances and the expected outcome.

The rules

The basic rules that govern how a merge is done are as follows:

Eyes never combines source and target images. The way to update baseline images so that they reflect the expected outcome of merged source and target code is to run a visual UI test on the combined code, view the result and accept the images that need resolution .

will always take images and annotations for a given step from the same baseline - either the source, or the target, and will never combined the annotations from two different baselines.

When eyes does a compare operation it can find that a particular baseline has no conflict, and can be combined automatically. This can only happen if the source branch was originally branched from the target branch and can identify steps that were in the original baseline and were changed in either the source or to the target (but not both). If the user does not explicitly request to use the source or the target , then these updated steps will be used in the merged target "as is".

If Eyes finds a conflict between the source and target, then you can either choose to take the source or the target. If you choose to take the source then you can specify whether the annotations are to be taken from the source, the target or some mix. The images will always be taken from the source in this case.
